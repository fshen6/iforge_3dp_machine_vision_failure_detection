## Problem Statement

The user operates a 10-printer 3D print farm. All printers are the same model, all run OctoPrint, all have cameras with consistent lighting. Today, print failures (bed adhesion, layer shift, spaghetti, clogs, warping) waste filament, time, and require manual babysitting to catch. There is no real-time failure detection. The goal is an AI vision system that watches every print, detects failures across the entire print duration, automatically pauses the affected printer via OctoPrint's API, and notifies the operator.

The user already has labeled training data: ~1000 successful print timelapses and hundreds of failed timelapses, with failure reason and approximate timestamp tagged on each failure. This is the asset that makes this project tractable.

## What Makes This Cool

- A working internal tool that actually pays for itself in saved filament and reclaimed operator attention. The "$200 of PLA wasted because nobody noticed for 8 hours" problem dies.
- The dataset advantage is rare. Most 3D printer vision projects (Obico, Spaghetti Detective) had to crowdsource failure data over years. The user already has it from operating a real farm. Day-one training, no cold-start.
- Homogeneous fleet (same printer, same camera, same lighting) means a single model transfers across all 10 machines with no per-printer calibration. This is the inverse of the usual "every customer's setup is different" problem that makes commercial vision products hard.
- The system is end-to-end controllable by one person: data, model, integration, deployment, all on the user's LAN. No vendors, no clouds, no contracts.

## Constraints

- **Internal tool only.** Not productized. Runs on user's LAN, used by the user.
- **OctoPrint integration required.** Pause/notify must go through OctoPrint's REST API. No other firmware in scope.
- **Local inference preferred.** Mini-PC or Raspberry Pi 5 on the LAN. No mandatory cloud dependency for v1.
- **Real-time enough.** A failure detected within 30–60 seconds of the visual onset is sufficient. Sub-second latency not needed.
- **Auto-pause + notify behavior.** Not just alerting — the system pauses the printer. False positives have a real cost (paused healthy print) but recoverable (operator hits resume).
- **No labeling work in v1.** The existing labeled dataset is the training set. Future data collection augments it, doesn't gate the v1 release.

## Premises (agreed)

1. The problem is "detect failure as early as possible on any print across its full duration" — not "build a generic failure classifier for the public." Scope is this user's farm, this user's printers, this user's data.
2. The user's labeled dataset of ~1000 healthy + hundreds of failure-reason-tagged timelapses removes the cold-start problem entirely. Multi-class supervised training is viable from day one.
3. Homogeneous fleet (same printer model, same cameras, same lighting per machine) means one model trained on this dataset transfers to all 10 printers without per-machine adaptation.
4. The hard part is not the vision model. It is the integration: camera-frame pipeline, inference service, OctoPrint pause API, notification delivery, and operator UI. The model is swappable behind a stable interface.
5. Distribution is trivial: a single mini-PC on the LAN runs the inference service, polls all 10 OctoPrint instances and their camera streams. No release engineering, no container registry, no app store.

## Approaches Considered

### Approach A: Single-frame binary classifier on first 10 min only (Minimal Viable)
Filter the dataset to first-10-min frames. Train a small ResNet/EfficientNet/ViT-Small for binary `healthy_first_layer / not_healthy`. One inference every 10s during first 10 min only. Rolling-window decision (3 of 5 frames flagged → pause).
- Effort: S (2–3 weeks end-to-end)
- Risk: Low — single label, small model, easy to debug
- Pros: ships fastest, easy to interpret, trivial inference infra, throws away nothing the user really needs in a first version
- Cons: ignores post-first-layer failures (spaghetti at hour 6, mid-print layer shifts), throws away most of the user's labeled failure data
- Reuses: timm/torchvision pretrained backbones, OctoPrint REST API, ntfy or Telegram for notifications

### Approach B: Multi-class CNN, full print duration (Chosen)
Use the full labeled dataset, including failures across the entire print. Classes: `healthy / spaghetti / bed_adhesion / layer_shift / other_failure` (exact class list TBD after dataset audit). Same backbone family as A, multi-class head. Run continuously across the entire print at fixed cadence (e.g., one frame every 10s).
- Effort: M (4–6 weeks)
- Risk: Med — more failure modes mean more confusion pairs and harder threshold tuning
- Pros: catches the high-value mid-print failures (spaghetti at hour 6 saves a 12-hour print), uses the full dataset, multi-class predictions can drive richer notifications ("looks like spaghetti on printer 4")
- Cons: more evaluation complexity, larger false-positive surface, longer time-to-first-ship than A
- Reuses: same as A plus the full labeled timelapse archive

### Approach C: Two-stage cascade — cheap classifier + VLM verifier
Stage 1: small fast classifier tuned for high recall flags suspicious frames. Stage 2: when stage 1 flags, send the frame plus 2–3 adjacent frames to a VLM (Claude or local Llama-Vision) with a structured prompt that confirms or rejects. Only pauses after stage 2 confirms.
- Effort: M (3–5 weeks)
- Risk: Med — adds API dependency (if cloud VLM) or GPU dependency (if local VLM), but only on the small percentage of flagged frames
- Pros: dramatically lower false-pause rate, natural-language failure explanations in notifications ("right corner is lifting"), recoverable: if VLM is down the system degrades to alert-only mode
- Cons: cloud or GPU dependency for stage 2, more components to maintain, per-failure verification cost (cents if cloud, free if local)
- Reuses: A's model as stage 1, Claude API or a local VLM as stage 2

## Recommended Approach

**Approach B: Multi-class CNN, full print duration.**

The user chose this over A because the dataset already covers the full print and discarding mid-print failures would discard most of the labeled data. The user has the asset; the design should use it. The risk that B becomes a tuning quagmire is real but manageable: start with the 3 highest-frequency classes from the dataset, ship binary `healthy / failure` thresholding first if multi-class proves noisy, and add classes iteratively.

A natural v1.5 extension is to layer Approach C's VLM verifier on top of B once real false-positive data exists. The user can keep that path open by designing the inference service so the model is a swappable component behind a single-frame `predict(image) -> {label, confidence}` interface.

### Implementation shape (high level, not a build plan)

1. **Dataset prep.** Audit the labeled timelapses, decode to frames at 10s cadence, build train/val/test splits with no print-level leakage. Concretely: assign each timelapse a unique print_id (filename hash or explicit ID), split at the print_id level, never the frame level. Per-printer stratification so each split has frames from each of the 10 printers.
2. **Model — two-stage architecture (per eng review Issue 3).** Stage 1: binary `healthy / failure` classifier (uses abundant data on both sides). Stage 2: multi-class head `spaghetti / bed_adhesion / layer_shift / other` only runs when stage 1 flags failure. Backbone: EfficientNet-B0 or ViT-Small. Track per-class precision, recall, time-to-detection.
3. **Eval set (per eng review Issue 4).** Separate from train/val/test. Hand-curate 50–100 frames per failure scenario from the dataset, including the hard cases (subtle bed-lift onset, low-light, edge cases). This is the gold benchmark for grading any future model.
4. **Inference service.** Single Python service on a mini-PC. Polls OctoPrint `/api/job` every 5s per printer to detect print-start transitions (Operational → Printing). On print-start, opens a monitoring window for that printer. Pulls camera snapshot every 10s during the window, runs inference, writes results + frame path to SQLite.
5. **SQLite schema (sketch).**
   ```
   frames(id, printer_id, print_id, timestamp, image_path, stage1_label, stage1_conf, stage2_label, stage2_conf, action_taken, operator_label, operator_responded_at)
   prints(id, printer_id, started_at, ended_at, end_reason, gcode_filename)
   ```
6. **Decision layer.** Rolling-window vote: 3 of last 5 frames flagged as `failure` at stage 1 → pause. Per-class confidence thresholds tuned against the eval set.
7. **Action layer.** Pause: call OctoPrint pause API with retry-with-backoff (3 attempts, 2s/5s/10s). If all fail, escalate to a "PAUSE FAILED" notification (per noted mitigation). Notification: ntfy with two inline action buttons — `Real failure` and `False alarm`. Button tap writes operator_label to SQLite (per Issue 2 labeling loop). Frame snapshot embedded.
8. **Shadow mode (per eng review Issue 5).** A `SHADOW_MODE=true` flag in config replaces the OctoPrint pause API call with a `log_what_would_have_happened()` write. Notifications still send. Run in this mode on all 10 printers for one week minimum before flipping to live.
9. **Frame retention (per eng review Issue 6).** Save every captured frame to disk for 90 days, then auto-delete unless the frame is flagged (stage 1 said failure) or labeled (operator tapped a button). Cron job nightly. **Storage budget at 10s cadence:** 10 printers × 1 frame/10s × 86400 s/day × ~100 KB ≈ **8.6 GB/day**, ≈ **775 GB at 90-day peak**. Provision a 1 TB external SSD for the inference host. (Compared to 30s cadence which would have been ~260 GB peak — the 3x storage cost is real but cheap.)
10. **Reliability mitigations.**
    - systemd unit with `Restart=always` for the inference service.
    - Heartbeat: service publishes "alive" to ntfy every 5 minutes; a separate watchdog on the mini-PC sends "service is dead" if no heartbeat for 10 min.
    - OctoPrint pause API failure escalates to "PAUSE FAILED, GO LOOK" notification after retries exhausted.
11. **Operator UI.** Still stretch — but the notification-with-buttons in step 7 covers the labeling-loop need without requiring a UI for v1.

## Hardware

### Inference host (primary): Raspberry Pi 5 (8 GB) + Hailo-8 AI HAT (26 TOPS)
- ~$80 Pi 5 8GB + ~$249 Hailo-8 AI HAT = ~$330 total
- 26 TOPS INT8 via Hailo-8. Handles YOLOv8n-cls at >300 fps; the system needs 1 fps total. ~300x headroom.
- Low power (~10 W total), passive cooling sufficient, GPIO available for future printer integration
- Connect a 1 TB external SSD via USB 3.0 for frame storage, OR use a Pi 5 PCIe NVMe HAT (note: cannot stack with AI HAT — pick one or use USB SSD)
- Toolchain tradeoff: Hailo Dataflow Compiler is less mature than NVIDIA's TensorRT. Model conversion (ONNX → HEF) requires an INT8 calibration set every time you compile. Plan for an extra step in the build pipeline; not hard, just real work.

### Inference host (fallback): Jetson Orin Nano 8GB
- ~$499 dev kit
- 40 TOPS INT8, TensorRT toolchain is more mature than Hailo's
- Use as fallback only if the Pi 5 + Hailo path hits a wall (e.g., a layer in your model doesn't compile to HEF and substitution isn't acceptable)

### Models
- **Stage 1 (binary, runs every 10s per printer):** YOLOv8n-cls. Picked over YOLOv11n-cls because YOLOv8 has been on the Hailo Model Zoo for over a year and compiles cleanly. YOLOv11 may work but adds risk for no real accuracy gain at this scale.
- **Stage 2 (multi-class, runs only when stage 1 flags):** YOLOv8s-cls or YOLOv8m-cls. Heavier variant OK because stage 2 fires rarely. The Hailo-8 has plenty of headroom.
- Both stages fine-tuned from ImageNet weights via Ultralytics. Do not train from scratch.
- **Backup plan if YOLOv8-cls compilation fails on Hailo:** use Hailo Model Zoo's pre-compiled MobileNetV2 or EfficientNet-Lite0 (`.hef` files ship ready to run, you only fine-tune the head). Strictly safer, slightly more wiring work.

## Training environment

Training is separate from inference. The Jetson/Pi is for inference only — too slow for training and you don't want to risk thermal throttling on production hardware.

### Where to train
Pick whichever of these you already have or can get to first:

1. **Local PC with a discrete NVIDIA GPU (preferred).** Anything from RTX 3060 (12 GB) upward. Fine-tuning EfficientNet-B0 on ~30K frames takes under an hour. Free if you have the hardware. Best for fast iteration during model development.
2. **Google Colab.** Free tier works for initial experiments (T4 GPU, time-limited). Colab Pro ($10/month) for serious training runs. Mount the dataset from Google Drive. Best if you don't have a GPU PC.
3. **RunPod / Vast.ai / Lambda Labs.** Pay-per-hour cloud GPUs. ~$0.50-0.80/hr for an RTX A4000 or A5000. A full v1 training run costs $1-3. Best for occasional retraining if you don't want a monthly Colab subscription.

Do NOT train on the Jetson Orin Nano. Inference target, not a training rig — 8 GB unified memory and limited cooling make it the wrong tool.

### Training pipeline (concrete recipe)

```
┌─ Dataset (~30K labeled frames) ─┐
│  good/ + spaghetti/ + bed_adhesion/ + layer_shift/ + other/
│  split by print_id, never by frame
└──────┬──────────────────────────┘
       │
       ▼
┌─ Stage 1 (binary) ────────────────┐         ┌─ Stage 2 (multi-class) ────────┐
│  EfficientNet-B0, ImageNet-pretrained │     │  ConvNeXt-Tiny, ImageNet-pretrained │
│  Output: healthy / failure            │     │  Output: spaghetti / adhesion / shift / other │
│  Trained on full dataset              │ ──> │  Trained on FAILURE frames only │
│  Class-weighted loss (healthy:failure)│     │  Class-weighted loss across types │
└──────┬────────────────────────────┘         └──────┬─────────────────────────┘
       │                                              │
       ▼                                              ▼
   Export ONNX                                    Export ONNX
       │                                              │
       └──────────────┬───────────────────────────────┘
                      ▼
              TensorRT INT8 conversion
                      │
                      ▼
              Deploy to Jetson Orin Nano
```

Training procedure (per stage, same recipe):
1. Load pretrained backbone from `timm.create_model('efficientnet_b0', pretrained=True)`.
2. Replace the classifier head with a new `nn.Linear(in_features, num_classes)`.
3. Freeze the backbone, train the head only for 5 epochs at lr=1e-3.
4. Unfreeze everything, train end-to-end for 10-15 epochs at lr=1e-4 with cosine LR schedule.
5. Apply augmentations: color jitter (saturation 0.3, brightness 0.3, contrast 0.3), small rotation (±10°), random crop. **No horizontal or vertical flip** — print orientation matters.
6. Use `CrossEntropyLoss(weight=class_weights)` where class_weights are inverse to class frequency.
7. Save the checkpoint with the best validation F1 (not accuracy — accuracy is misleading with class imbalance).
8. Grade the final checkpoint against the curated eval set. v1 needs to beat baseline thresholds before deployment.

Total wall-clock time on an RTX 3060: ~45 min per stage. Two stages = ~90 min per full training run. Fast enough to iterate hyperparameters in an afternoon.

### Deployment

```bash
# On training machine after training completes:
# Option A: Ultralytics one-shot export to Hailo HEF
yolo export model=runs/stage1_binary/weights/best.pt format=hailo data=calib_frames/

# Option B: explicit ONNX → Hailo Dataflow Compiler path (more control)
yolo export model=runs/stage1_binary/weights/best.pt format=onnx imgsz=224
hailo parser onnx best.onnx
hailo optimize best.har --calib-set-path calib_frames/
hailo compile best_optimized.har --hw-arch hailo8
# produces best.hef

# Copy compiled engines to the Pi:
scp stage1.hef stage2.hef pi@inference-host:/opt/print-vision/models/

# On the Pi 5, the inference service loads .hef files via HailoRT bindings.
systemctl restart print-vision
```

Hailo is **INT8-only**. The calibration set is mandatory — feed the compiler ~500 representative frames (mix of healthy + each failure class) and it tunes the per-layer quantization scales. Use frames from your training set's healthy class plus a handful from each failure class. Skipping this step gives broken accuracy.

### Retraining cadence

- **v1:** train once on the existing labeled dataset.
- **v2 and on:** the labeling-loop button taps (Real failure / False alarm) accumulate operator-labeled data over weeks. Retrain when you have ~500+ new operator labels, or quarterly, whichever comes first.
- Each retraining run is the same 90-minute recipe. ONNX export, TensorRT recompile, drop in. Compare against curated eval set before promoting. v2 must beat v1 on every class to ship.

## Next Steps (build order)

1. **Dataset audit.** Open the labeled dataset, count examples per failure class, check frame rates, check whether failure timestamps are precise to layer/minute. Output a one-page audit report with per-class counts.
2. **Curate eval set.** Hand-pick 50–100 frames per failure scenario (healthy + each failure class). Include the hard cases. Lock this set — never train against it.
3. **Frame extraction + print-level splits.** Decode timelapses to frames at 30s cadence. Build train/val/test splits at the `print_id` level. Stratify by printer.
4. **Stage 1 binary model.** Fine-tune EfficientNet-B0 on healthy/failure. Track per-class precision/recall against the eval set.
5. **Stage 2 multi-class head.** Train classification head on failure-only frames. Same backbone, frozen features.
6. **Inference service skeleton.** Python service that polls OctoPrint `/api/job` every 5s, detects print-start, pulls snapshots every 10s, writes to SQLite. No model yet. Validate camera + OctoPrint pipeline on one printer.
7. **Wire model + notifications + labeling buttons.** Stage 1 model in service. ntfy notifications with `Real failure` / `False alarm` action buttons. Operator taps land in SQLite.
8. **Shadow mode on all 10 printers, one week minimum.** `SHADOW_MODE=true`. Service runs everywhere, predicts everywhere, notifies everywhere, never actually pauses. Tune thresholds against measured false-positive rate.
9. **Flip to live on one printer for one week.** First production printer with auto-pause enabled. Watch closely.
10. **Roll to all 10.** Add heartbeat watchdog. Set up 90-day frame retention cron. Start using accumulated labeled data for v2 training.

## Failure modes (production scenarios + mitigations)

| Scenario | Mitigation | Test it? |
|---|---|---|
| Camera returns blank frame / 404 | Skip the frame, log, alert if >5 in a row | yes — mock OctoPrint snapshot endpoint |
| OctoPrint API down when pause triggered | Retry with backoff (2s/5s/10s), then "PAUSE FAILED" notification | yes — mock OctoPrint to 500 |
| Inference service crashes mid-print | systemd auto-restart + heartbeat watchdog | yes — kill the process, verify recovery |
| Mini-PC power blip | systemd starts on boot, state recovers from SQLite | yes — power-cycle test |
| All 10 printers print-start simultaneously | Service handles concurrent windows (10 parallel poll loops, not a queue) | yes — load test |
| Phone offline when notification fires | ntfy stores undelivered notifications, delivers on reconnect | manual smoke test |
| Operator never taps the label buttons | Label is `null` in SQLite, doesn't break anything, just no signal for retraining | n/a — graceful by design |
| Class drift over months (new filament, dust, lighting) | Eval set + quarterly retraining cycle | n/a — operational, not testable upfront |

**Critical gap flagged:** the "phone offline" scenario means the operator can miss a real failure entirely if their phone is dead AND they're not watching the farm. This is a v2 concern (consider a webhook to a fallback service / loud alarm on the mini-PC), not a v1 blocker.

## Worktree parallelization strategy

Sequential implementation, no meaningful parallelization opportunity. The build order is genuinely linear: dataset → model → service → labeling → shadow → live. One person, one mini-PC, one repo. No parallel lanes worth splitting.
